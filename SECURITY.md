# Security Policy

FluxFormula is maintained by a solo developer. This document describes how to report security issues and what falls within the project's security scope.

## Reporting a Vulnerability

If you discover a security vulnerability, please report it privately via GitHub's security advisory system:

> https://github.com/twds0x13/FluxFormula/security/advisories/new

Please include:

- A description of the vulnerability and its impact
- Steps to reproduce, including a minimal code sample if applicable
- The affected version(s)
- Any known workarounds

I will acknowledge the report as soon as I can. As a solo-maintained project, there is no fixed SLA, but security issues are treated as high priority.

**Do not open a public issue for security vulnerabilities.**

## Supported Versions

Only the latest release receives active security fixes. If a vulnerability is reported against an older version, the fix will be made on the current release line. Users are encouraged to stay on the latest version.

## Threat Model

FluxFormula is a Unity package that compiles and executes user-defined formulas at runtime. The following considerations inform our security assessment:

### In Scope

- **Malicious formula input causing unintended code execution**: The JIT path compiles expressions via `System.Linq.Expressions`. Only expressions generated by user-provided `IFluxJITDefinition.GetExpression()` are compiled — FluxFormula itself does not emit arbitrary IL.
- **Buffer overflows via crafted bytecode**: `Instruction[]` buffers are accessed via `fixed` pointers. Invalid offsets or register indices could theoretically access out-of-bounds memory. FluxFormula validates register bounds in debug builds; release builds trust the compiler output.
- **Denial of service via resource exhaustion**: Extremely long formulas or deeply nested expressions could exhaust the operator stack (max 64 levels) or register pool (253 general-purpose registers). These limits serve as built-in DoS mitigations.

### Out of Scope

- **Unity Editor or runtime vulnerabilities unrelated to FluxFormula**: FluxFormula runs within Unity's managed runtime and does not bypass Unity's security model.
- **User-provided `IFluxJITDefinition` implementations**: The definition interface is extensible by design. Security of `GetExpression()` and `Compute()` implementations is the responsibility of the integrating project.
- **Input validation of formula strings**: The Lexer parses strings but does not sanitize for injection attacks — formula strings are assumed to come from trusted sources (developers or build pipelines).

### Best Practices for Integrators

- Do not pass unsanitized user input directly to `FluxLexer.Lex()`.
- Validate `IFluxJITDefinition` implementations from third-party sources before use.
- Use the interpreter path (`jit: false`) in contexts where untrusted formula data is processed, as it avoids `Expression.Compile()` entirely.
- Keep FluxFormula updated to the latest version.
